# PC 端本地文字转语音桌面应用方案（基于 Coqui TTS）

## 1. 方案目标

本方案用于构建一个运行在 **PC 端本地** 的文字转语音桌面应用，采用如下技术路线：

- **Tauri**：桌面应用框架
- **Vue / React 前端页面**：负责 UI 交互
- **Rust**：负责 Tauri 命令桥接、文件管理、进程管理
- **Python 本地服务**：负责语音合成与声音克隆
- **Coqui TTS**：作为核心 TTS / Voice Cloning 引擎

目标能力包括：

- 本地输入文本并生成音频
- 支持多个内置音色
- 支持用户克隆自己的声音
- 支持离线运行
- 支持本地播放和导出音频文件
- 支持后续扩展批量生成、语速调节、多语言等功能

---

## 2. 核心需求拆解

本项目的核心需求包括：

### 1）文字转音频
输入一段文本，系统在本地生成语音音频文件。

### 2）内置多个音色
应用提供若干预置音色，例如：

- 女声 1
- 女声 2
- 男声 1
- 男声 2
- 中性旁白

### 3）支持声音克隆
用户上传或录制一段自己的声音样本，系统生成一个可复用的个人音色 profile。

### 4）本地离线运行
模型推理、音频生成、profile 管理都在本机完成，不依赖远程 API。

## 4. 为什么选择 Python + Coqui TTS

相较于直接用 Rust 原生实现推理，当前 TTS / 声音克隆领域更适合采用 Python 路线，原因包括：

- Coqui TTS 的主生态在 Python
- 模型支持更成熟
- 声音克隆能力更容易接入
- 音频处理工具链更完善
- 后续扩展训练、微调、embedding 提取更方便

因此推荐架构：

```text
前端 -> Tauri Rust -> Python 本地服务 -> Coqui TTS
```

---

## 5. 为什么选择 Coqui TTS

选择 Coqui TTS 的原因：

- Python 生态成熟
- 支持多说话人 TTS
- 支持声音克隆相关能力
- 社区资料较多
- 适合本地部署和桌面应用集成
- 可作为后续升级到更复杂语音方案的基础

适合本项目的目标：

- 内置多个音色
- 用户自定义声音
- 离线本地运行
- 后期扩展多语言

## 8. 功能模块设计

### 1）文本转语音模块

输入：

- 文本
- 音色 ID
- 语速
- 语言
- 输出格式

输出：

- 音频文件路径
- 或生成任务结果

---

### 2）内置音色模块

建议预置若干 speaker profile：

- `female_01`
- `female_02`
- `male_01`
- `male_02`
- `narrator_01`

实现方式可以是：

- 使用 Coqui 模型支持的 speaker
- 使用预定义参考音频

输入：

处理过程：

- 提取说话人特征
- 生成 voice profile
- 保存 profile 供后续复用

输出：

- `voiceProfileId`
- profile 元信息

---

### 4）音频播放与导出模块

支持：

- 本地试听
- 重新生成
- 导出 `.wav`
- 后续可扩展 `.mp3`

---

### 5）任务管理模块

建议维护任务状态：

- `pending`
- `processing`
- `success`
- `failed`

## 9. 数据流设计

### 普通 TTS 流程

```text
前端输入文本
   ↓
选择内置音色
   ↓
调用 Tauri command
   ↓
Rust 转发给 Python 本地服务
   ↓
Python 调用 Coqui TTS 生成音频
   ↓
返回音频文件路径
   ↓
前端播放 / 导出
```

### 声音克隆流程

```text
用户上传或录制参考音频
   ↓
前端调用 clone_voice
   ↓
Rust 转发给 Python 本地服务
   ↓
Python 处理参考音频并生成 voice profile
   ↓
保存 profile
   ↓
返回 profile ID
   ↓
后续生成时可直接使用该 profile
```

## 10. 核心接口设计

### `generate_speech`

输入示例：

```json
{
  "text": "你好，欢迎使用本地语音合成应用。",
  "voiceId": "female_01",
  "speed": 1.0,
  "language": "zh",
  "outputFormat": "wav"
}
```

输出示例：

```json
{
  "taskId": "tts-001",
  "status": "success",
  "audioPath": "/path/to/output.wav"
}
```

---

### `clone_voice`

输入示例：

```json
{
  "name": "my_voice",
  "audioPath": "/path/to/reference.wav",
  "language": "zh"
}
```

输出示例：

```json
{
  "voiceProfileId": "voice-user-001",
  "status": "success"
}
```

---

### `list_voices`

返回：

- 内置音色列表
- 用户克隆音色列表

---

### `delete_voice_profile`

删除某个用户自定义音色。

## 11. 内置音色设计建议

内置音色应设计为真正可展示的 profile，而不只是字符串：

```json
{
  "id": "female_01",
  "name": "温柔女声",
  "type": "builtin",
  "language": ["zh", "en"],
  "previewAudio": "/assets/builtin-voices/female_01_preview.wav"
}
```

这样前端可以：

- 展示名称
- 支持试听
- 区分内置 / 用户克隆
- 扩展语言标签和分类标签

## 12. 声音克隆实现思路

### MVP 推荐方案
第一版不建议做“重新训练一个完整模型”，而建议做：

**参考音频驱动的 voice cloning / speaker profile 方案**

即：

- 用户提供参考音频
- 系统提取音色特征
- 保存为 voice profile
- 后续生成时以该 profile 为参考

优点：

- 资源占用更小
- 实现复杂度更低
- 更适合桌面应用
- 更适合 MVP 落地

---

## 13. 用户录音要求建议

为了提高克隆质量，建议前端明确提示用户：

- 录音时长建议 10～60 秒
- 环境尽量安静
- 仅单人说话
- 不要有背景音乐
- 尽量使用高质量音频

推荐格式：

- `.wav`
- 单声道优先
- 中高质量采样率均可，后端统一处理

---

## 14. 音频预处理建议

后端建议统一做以下预处理：

- 重采样
- 音量归一化
- 去除异常静音（可选）
- 降噪（可选）

预处理的目标是：

- 提高克隆质量
- 提高生成稳定性
- 统一不同来源音频格式

## 21. 风险点

### 1）声音克隆质量依赖参考音频
如果参考音频太短、噪声太大，效果会明显下降。

### 2）模型体积较大
本地 TTS / voice cloning 模型会影响首次启动时间和安装包大小。

### 3）打包复杂度较高
因为涉及：

- Tauri
- Rust
- Python
- 模型文件
- 音频依赖库

### 4）跨平台差异
Windows、macOS、Linux 下音频设备、路径、依赖可能不同。
